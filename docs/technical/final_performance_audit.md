### BOTTLENECKS IDENTIFIED

#### 1. Monolithic JSON Fetching (`summaries.json`) — Impact: HIGH
*   **Location**: `js/app.js` -> `renderSubjectPage()`
*   **Evidence**: The entire `summaries.json` (currently 237KB, projected 1.2MB+) is fetched on every subject-specific page.
*   **Root cause**: Filtering is done client-side after a full download of the knowledge base.
*   **Fix**: 
    *   **Vertical Partitioning**: Split `summaries.json` into `summaries_[subject].json`.
    *   **Lazy Loading**: Only fetch the summary content when the user clicks "อ่านสรุปบทเรียน" or use a lightweight index for the list view.
*   **Expected improvement**: 70-90% reduction in data transfer for subject pages.

#### 2. Render-Blocking Third-Party Assets — Impact: MED
*   **Location**: `index.html` (Google Fonts, FontAwesome, Chart.js)
*   **Evidence**: 4+ external requests in the `<head>` block the First Contentful Paint.
*   **Root cause**: Standard script/link tags without `async`, `defer`, or critical path optimization.
*   **Fix**: 
    *   Move `Chart.js` to a deferred script or load it dynamically only when a chart needs to be rendered.
    *   Inline critical CSS for the Hero section and "Subject Grid" layout.
*   **Expected improvement**: ~400ms reduction in FCP.

#### 3. PWA Cache Strategy Optimization — Impact: MED
*   **Location**: `sw.js`
*   **Evidence**: The service worker caches the full `summaries.json` on install. If this file grows to 1.2MB, the initial install/update will be heavy on mobile data.
*   **Fix**: Use a `Cache-First` or `Stale-While-Revalidate` strategy for data files, but avoid pre-caching the entire knowledge base. Only cache on-demand.
